# 协作工作流程

## 📋 工作流程概述

我们采用 **本地开发 + Git 协作 + Linux 服务器部署** 的工作模式：

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   本地开发      │ -> │   Git 推送      │ -> │ Linux 服务器    │
│   (Trae AI)     │    │   (GitHub)      │    │   (生产环境)    │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

## 🔄 详细流程

### 阶段一：本地开发与验证

#### 1. 功能开发
- Trae AI 编写代码
- 遵循现有代码规范和架构
- 使用 TypeScript/Python 等语言特性

#### 2. 本地验证
**前端验证**：
```bash
cd frontend
npm run dev          # 启动开发服务器
npm run build        # 构建验证
npm run typecheck    # 类型检查
```

**后端验证**：
```bash
cd backend
python3 -m uvicorn main:app --reload  # 启动开发服务器
```

**数据库验证**：
```bash
python3 scripts/init_dev_db.py  # 初始化开发数据库
```

#### 3. 测试要点
- ✅ 功能正常运行
- ✅ API 接口返回正确数据
- ✅ 前端页面渲染正常
- ✅ 无控制台错误
- ✅ 数据库操作正常

### 阶段二：Git 提交

#### 1. 创建功能分支
```bash
git checkout -b feature/<功能名称>
```

#### 2. 提交代码
```bash
git add <修改的文件>
git commit -m "feat: <功能描述>"
```

#### 3. 推送到远程
```bash
git push -u origin feature/<功能名称>
```

#### 4. 分支命名规范
- `feature/xxx` - 新功能开发
- `fix/xxx` - Bug 修复
- `refactor/xxx` - 代码重构
- `docs/xxx` - 文档更新

### 阶段三：Linux 服务器部署

#### 1. 拉取代码
```bash
git pull origin feature/<功能名称>
# 或合并到主分支
git checkout main
git merge feature/<功能名称>
```

#### 2. 构建项目
**前端构建**：
```bash
cd frontend
npm install
npm run build
```

**后端构建**：
```bash
cd backend
pip install -r requirements.txt
```

#### 3. 配置环境
- 更新 `.env` 配置文件
- 配置数据库连接
- 配置 AI 服务 API Key

#### 4. 启动服务
```bash
# 使用 systemd 或其他进程管理工具
sudo systemctl restart my-open-calender-backend
sudo systemctl restart my-open-calender-frontend
```

#### 5. 验证部署
- 检查服务状态
- 访问生产环境 URL
- 测试关键功能

## 📝 提交信息规范

### 格式
```
<类型>: <描述>

[可选的详细说明]
```

### 类型说明
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建/工具链更新

### 示例
```
feat: 优化古诗推荐功能，支持根据关键词智能推荐

- 后端 ai_service.py: 改进 generate_poetry 方法，支持 keyword 参数
- 后端 routes.py: 修改 /poetry/chat 接口传递用户关键词
- 前端 PoetryChat.vue: 修复调用 chat API 并传递用户输入
```

## 🚨 注意事项

### 本地开发
- ✅ 完成功能后必须进行本地验证
- ✅ 确保没有控制台错误
- ✅ 测试关键功能路径
- ❌ 不要直接修改生产环境代码

### Git 提交
- ✅ 每个功能使用独立分支
- ✅ 提交信息清晰描述变更
- ✅ 提交前检查是否有遗漏文件
- ❌ 不要在主分支直接开发

### Linux 部署
- ✅ 部署前先备份数据库
- ✅ 使用测试环境验证
- ✅ 部署后监控服务状态
- ❌ 不要在业务高峰期部署

## 🔧 常用命令速查

### Git 操作
```bash
# 查看状态
git status

# 查看分支
git branch -a

# 切换分支
git checkout <分支名>

# 拉取最新代码
git pull origin <分支名>

# 查看提交历史
git log --oneline -10
```

### 服务操作
```bash
# 查看服务状态
sudo systemctl status <服务名>

# 重启服务
sudo systemctl restart <服务名>

# 查看日志
sudo journalctl -u <服务名> -f
```
