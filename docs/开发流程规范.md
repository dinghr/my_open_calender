# 开发流程规范

**创建日期：** 2026-03-12  
**目标：** 确保开发质量，避免返工

---

## ⚠️ 核心原则

**必须严格遵守的开发流程：**

```
需求细化 → 高保真原型 → 页面渲染确认 → 代码开发
   ↓           ↓            ↓           ↓
  Plan 模式   视觉确认      交互确认     实现代码
```

**禁止：** 跳过任何步骤直接进入代码开发！

---

## 📋 标准开发流程

### 阶段 1：Plan 模式 - 需求细化 ⭐⭐⭐⭐⭐

**目标：** 把需求磨得特别细，磨得特别细

**必须完成的事项：**

#### 1.1 功能需求清单
- [ ] 每个功能点的详细描述
- [ ] 用户故事（User Story）
- [ ] 使用场景说明
- [ ] 边界条件定义
- [ ] 异常处理逻辑

#### 1.2 数据结构设计
- [ ] 数据库表结构
- [ ] 字段类型和约束
- [ ] 表关系定义
- [ ] API 接口设计
- [ ] 数据流转图

#### 1.3 用户流程
- [ ] 用户操作流程图
- [ ] 页面跳转逻辑
- [ ] 状态变化说明
- [ ] 权限控制规则

#### 1.4 技术选型
- [ ] 前端框架
- [ ] 后端框架
- [ ] 数据库选择
- [ ] 第三方服务
- [ ] 部署方案

#### 1.5 验收标准
- [ ] 功能验收标准（Definition of Done）
- [ ] 性能指标
- [ ] 兼容性要求
- [ ] 安全要求

**输出文档：**
- `需求文档.md`
- `技术设计.md`
- `API 文档.md`
- `数据字典.md`

**完成标志：** 所有参与人员对需求理解一致，无歧义

---

### 阶段 2：高保真原型设计 ⭐⭐⭐⭐⭐

**目标：** 可视化确认，避免理解偏差

**必须完成的事项：**

#### 2.1 原型工具选择
推荐工具：
- **v0.dev** - AI 生成（首选，快速）
- **Figma** - 专业设计
- **墨刀** - 国产工具
- **Axure** - 复杂交互

#### 2.2 原型内容
- [ ] 所有页面布局
- [ ] 完整交互逻辑
- [ ] 状态变化（hover、点击、加载等）
- [ ] 弹窗和提示
- [ ] 表单验证
- [ ] 空状态展示
- [ ] 错误状态展示
- [ ] 响应式适配（移动端/桌面端）

#### 2.3 设计规范
- [ ] 配色方案（主色、辅色、强调色）
- [ ] 字体规范（字号、字重、行高）
- [ ] 间距规范（8px 倍数原则）
- [ ] 圆角规范
- [ ] 阴影规范
- [ ] 图标风格

#### 2.4 原型评审
- [ ] 产品负责人确认
- [ ] 开发人员确认技术可行性
- [ ] 最终用户代表确认（如有）
- [ ] 修改意见收集
- [ ] 原型迭代优化

**输出文档：**
- `高保真原型.html` 或 `Figma 链接`
- `设计规范.md`
- `交互说明.md`

**完成标志：** 所有参与人员签字确认原型

---

### 阶段 3：页面渲染确认 ⭐⭐⭐⭐

**目标：** 确认每个页面的最终效果

**必须完成的事项：**

#### 3.1 页面清单
- [ ] 列出所有需要开发的页面
- [ ] 每个页面的 URL 规划
- [ ] 页面优先级排序

#### 3.2 页面渲染
- [ ] 使用原型工具导出静态页面
- [ ] 或用 v0.dev 生成可交互页面
- [ ] 部署到测试环境
- [ ] 生成可访问的 URL

#### 3.3 页面评审
- [ ] 逐页确认布局
- [ ] 逐页确认交互
- [ ] 逐页确认文案
- [ ] 收集修改意见
- [ ] 迭代优化

#### 3.4 页面冻结
- [ ] 所有页面确认无误
- [ ] 标记为"已冻结"状态
- [ ] 后续修改需走变更流程

**输出文档：**
- `页面清单.md`
- `页面评审记录.md`
- `页面冻结确认书.md`

**完成标志：** 所有页面渲染确认，签字冻结

---

### 阶段 4：代码开发 ⭐⭐⭐

**目标：** 按照确认的原型和规范实现代码

**必须完成的事项：**

#### 4.1 开发准备
- [ ] 搭建开发环境
- [ ] 初始化项目脚手架
- [ ] 配置代码规范（ESLint、Prettier）
- [ ] 配置 Git 分支策略
- [ ] 准备测试数据

#### 4.2 组件开发
- [ ] 开发基础组件（按钮、输入框等）
- [ ] 开发业务组件（任务卡片、成长树等）
- [ ] 编写组件文档
- [ ] 组件单元测试

#### 4.3 页面开发
- [ ] 按照原型逐页开发
- [ ] 实现页面交互逻辑
- [ ] 对接后端 API
- [ ] 页面自测

#### 4.4 代码审查
- [ ] 提交 Pull Request
- [ ] 团队成员 Code Review
- [ ] 修改审查意见
- [ ] 合并到主分支

#### 4.5 测试验收
- [ ] 功能测试
- [ ] 兼容性测试
- [ ] 性能测试
- [ ] 安全测试
- [ ] Bug 修复

**输出文档：**
- `代码仓库`
- `组件文档.md`
- `测试报告.md`
- `部署文档.md`

**完成标志：** 通过所有测试，可以上线

## 🚫 禁止行为

### ❌ 绝对禁止

1. **跳过 Plan 模式直接开发**
   - 后果：需求理解偏差，大量返工

2. **没有高保真原型就写代码**
   - 后果：边做边改，效率极低

3. **页面未确认就进入开发**
   - 后果：反复修改，延期交付

4. **口头需求直接开发**
   - 后果：需求不清晰，责任不明

5. **开发过程中随意变更需求**
   - 后果：项目失控，无法交付

### ⚠️ 需要审批的变更

以下变更需要产品负责人审批：
- 功能增删
- 页面布局重大调整
- 技术方案变更
- 工期超过 1 天的修改

## 🎯 各阶段时间分配建议

| 阶段 | 时间占比 | 说明 |
|-----|---------|------|
| Plan 模式（需求细化） | 30% | 磨得特别细 |
| 高保真原型 | 25% | 视觉确认 |
| 页面渲染确认 | 15% | 交互确认 |
| 代码开发 | 30% | 实现代码 |

**关键：** 前期多花时间，后期少返工

**错误做法：** 需求 1 天，开发 10 天（后期大量返工）

**正确做法：** 需求 + 原型 7 天，开发 10 天（一次性做对）

## 🎓 为什么这个流程重要？

### 传统开发的问题

```
需求（模糊） → 开发（猜测） → 测试（发现问题） → 返工（大量）
                ↓
            理解偏差
                ↓
            推倒重来
```

**结果：** 80% 的时间在返工

### Plan 模式的优势

```
需求（超详细） → 原型（可视化） → 确认（无歧义） → 开发（一次性）
       ↓              ↓              ↓              ↓
    磨得细         看得见         签字确认       照图施工
```

**结果：** 90% 的工作一次性做对

## 💡 实际案例

### 案例：AI 伴学小程序 - 成长树功能

**错误做法（实际发生过）：**
```
1. 口头说"要一个成长树"
2. 直接开始写代码
3. 做完发现不是想要的
4. 推倒重来
5. 反复 3 次
6. 浪费 2 周时间
```

**正确做法（应该这样）：**
```
1. Plan 模式（2 天）
   - 成长树的等级设计
   - 经验值计算规则
   - 视觉效果描述
   - 动画效果说明

2. 高保真原型（1 天）
   - v0.dev 生成 3 个版本
   - 选择最喜欢的
   - 调整细节

3. 页面渲染（0.5 天）
   - 部署可交互原型
   - 确认效果

4. 代码开发（3 天）
   - 按照原型实现
   - 一次性通过
```

**总计：** 6.5 天 vs 错误做法的 10 天 + 3 次返工

## 📞 流程执行监督

### 谁负责监督？

- **产品负责人：** 确认需求和原型
- **技术负责人：** 确认技术可行性
- **项目经理：** 监督流程执行

### 如何确保执行？

1. **每个阶段结束必须签字确认**
2. **没有签字不能进入下一阶段**
3. **变更必须走审批流程**
4. **定期检查流程执行情况**

## 🎯 总结

**核心思想：**

> **磨刀不误砍柴工**

**行动指南：**

1. ✅ 需求一定要磨得特别细
2. ✅ 一定要有高保真原型
3. ✅ 一定要页面渲染确认
4. ✅ 一定要签字冻结
5. ✅ 然后才能开始写代码

**违反后果：**

> 返工、延期、质量差、团队疲惫

**遵守收益：**

> 高效、高质量、按时交付、团队轻松
