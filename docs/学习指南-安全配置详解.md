## 第二部分：SSL/TLS 配置

### 2.1 什么是 SSL/TLS？

**SSL (Secure Sockets Layer)** 和 **TLS (Transport Layer Security)** 是加密协议，用于：
- 🔒 加密客户端和服务器之间的通信
- 🔐 验证服务器身份
- 🛡️ 防止中间人攻击

**版本演进：**
```
SSL 2.0 (1995) ❌ 已废弃
SSL 3.0 (1996) ❌ 已废弃
TLS 1.0 (1999) ❌ 已废弃
TLS 1.1 (2006) ❌ 已废弃
TLS 1.2 (2008) ✅ 可用
TLS 1.3 (2018) ✅ 推荐
```

---

### 2.2 我们的 SSL 配置

**文件：** `deploy/nginx-prod.conf`

```nginx
# 只允许 TLS 1.2 和 1.3
ssl_protocols TLSv1.2 TLSv1.3;

# 强加密套件
ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
ssl_prefer_server_ciphers off;

# SSL 会话缓存
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 1d;
ssl_session_tickets off;

# OCSP Stapling（提升性能）
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;
resolver_timeout 5s;
```

**安全分析：**

| 配置项 | 我们的设置 | 安全等级 | 说明 |
|--------|-----------|---------|------|
| 协议版本 | TLS 1.2+ | ✅ 优秀 | 禁用旧版本 |
| 加密套件 | ECDHE + AES-GCM | ✅ 优秀 | 前向保密 + 认证加密 |
| 会话缓存 | 10MB | ✅ 良好 | 提升性能 |
| OCSP Stapling | 开启 | ✅ 优秀 | 减少延迟 |

## 第三部分：HSTS 配置

### 3.1 什么是 HSTS？

**HSTS (HTTP Strict Transport Security)** 强制浏览器只通过 HTTPS 访问网站。

**工作原理：**
```
1. 浏览器首次访问 → HTTPS
2. 服务器返回 HSTS 头
3. 浏览器记住 HSTS 策略
4. 后续访问自动转 HTTPS（即使输入 http://）
```

---

### 3.2 我们的 HSTS 配置

**生产环境：**
```nginx
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
```

**参数说明：**
- `max-age=63072000` - 有效期 2 年（63072000 秒）
- `includeSubDomains` - 应用到所有子域名
- `preload` - 允许加入浏览器 preload 列表

**测试环境：**
```nginx
add_header Strict-Transport-Security "max-age=31536000" always;
```
- `max-age=31536000` - 有效期 1 年

---

### 3.3 HSTS 安全分析

| 特性 | 生产环境 | 测试环境 | 安全等级 |
|------|---------|---------|---------|
| max-age | 2 年 | 1 年 | ✅ 优秀 |
| includeSubDomains | ✅ | ❌ | ⚠️ 建议添加 |
| preload | ✅ | ❌ | ⚠️ 测试环境不需要 |

**建议改进：**
```nginx
# 测试环境也加上 includeSubDomains
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

## 第四部分：安全头配置

### 4.1 我们配置的安全头

```nginx
# 防止点击劫持
add_header X-Frame-Options "SAMEORIGIN" always;

# 防止 MIME 类型嗅探
add_header X-Content-Type-Options "nosniff" always;

# XSS 防护
add_header X-XSS-Protection "1; mode=block" always;

# 控制 Referrer 信息
add_header Referrer-Policy "strict-origin-when-cross-origin" always;

# 内容安全策略（可选，建议添加）
# add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline';" always;
```

---

### 4.2 安全头详解

#### X-Frame-Options

**作用：** 防止页面被嵌入 iframe（点击劫持攻击）

**选项：**
- `DENY` - 不允许任何网站嵌入
- `SAMEORIGIN` - 只允许同源网站嵌入 ✅ 我们的选择
- `ALLOW-FROM uri` - 允许指定网站嵌入

**攻击示例：**
```html
<!-- 攻击者网站 -->
<iframe src="https://api.ai-study.com" style="opacity:0"></iframe>
<!-- 用户在不知情的情况下点击 -->
```

**防护效果：** ✅ 阻止

---

#### X-Content-Type-Options

**作用：** 防止浏览器嗅探 MIME 类型

**为什么需要：**
```
服务器返回：Content-Type: text/plain
浏览器嗅探：这是 HTML，按 HTML 渲染
结果：XSS 攻击成功 ❌

启用 nosniff 后：
浏览器严格遵守：Content-Type: text/plain
结果：安全 ✅
```

**防护效果：** ✅ 阻止 MIME 混淆攻击

---

#### X-XSS-Protection

**作用：** 启用浏览器内置 XSS 过滤器

**注意：** 现代浏览器已弃用，但保留无坏处

**更好的方案：** Content-Security-Policy (CSP)

---

#### Referrer-Policy

**作用：** 控制发送 Referrer 头的策略

**我们的选择：** `strict-origin-when-cross-origin`

**行为：**
- 同源请求：发送完整 URL
- 跨域 HTTPS→HTTPS：只发送域名
- HTTPS→HTTP：不发送 Referrer

**隐私保护：** ✅ 良好

## 第五部分：CORS 配置

### 5.1 什么是 CORS？

**CORS (Cross-Origin Resource Sharing)** 控制网页能否访问其他域名的资源。

**为什么需要：**
```
浏览器同源策略：
- 页面 A (https://a.com) 不能访问 页面 B (https://b.com) 的数据
- 除非 B 明确允许（CORS 头）
```

---

### 5.2 我们的 CORS 配置

**生产环境 (backend/main.py)：**
```python
if ENV == "production":
    # 生产环境限制域名
    allowed_origins = os.getenv("ALLOWED_ORIGINS", "https://ai-study.com").split(",")
else:
    # 开发环境允许所有
    allowed_origins = ["*"]

app.add_middleware(
    CORSMiddleware,
    allow_origins=allowed_origins,  # 允许的源
    allow_credentials=True,          # 允许携带 Cookie
    allow_methods=["*"],             # 允许的方法
    allow_headers=["*"],             # 允许的头部
)
```

**配置说明：**

| 参数 | 生产环境 | 开发环境 | 说明 |
|------|---------|---------|------|
| allow_origins | 白名单 | * | 生产环境必须限制 |
| allow_credentials | True | True | 允许携带认证信息 |
| allow_methods | * | * | 允许所有 HTTP 方法 |
| allow_headers | * | * | 允许所有头部 |

---

### 5.3 CORS 安全分析

**当前配置：**
- ✅ 生产环境使用白名单
- ✅ 允许 credentials（需要认证）
- ⚠️ 允许所有方法和头部（方便但不够严格）

**建议改进：**
```python
# 更严格的配置
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://ai-study.com", "https://www.ai-study.com"],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Content-Type", "Authorization"],
)
```

## 第六部分：防火墙配置

### 6.1 UFW 防火墙

**UFW (Uncomplicated Firewall)** 是 Ubuntu 的防火墙工具。

**我们的配置：**
```bash
# 允许 SSH
sudo ufw allow 22/tcp comment "SSH"

# 允许 HTTP（用于 SSL 证书续期）
sudo ufw allow 80/tcp comment "HTTP"

# 允许 HTTPS（生产）
sudo ufw allow 443/tcp comment "HTTPS (Production)"

# 允许 HTTPS（测试）
sudo ufw allow 8443/tcp comment "HTTPS (Test)"

# 拒绝后端直接访问
sudo ufw deny 8001/tcp comment "Backend Prod (Nginx only)"
sudo ufw deny 8002/tcp comment "Backend Test (Nginx only)"

# 启用防火墙
sudo ufw enable
```

---

### 6.2 查看防火墙状态

```bash
# 查看状态
sudo ufw status verbose

# 输出示例：
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), deny (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere                   # SSH
80/tcp                     ALLOW IN    Anywhere                   # HTTP
443/tcp                    ALLOW IN    Anywhere                   # HTTPS
8443/tcp                   ALLOW IN    Anywhere                   # Test HTTPS
```

---

### 6.3 防火墙安全分析

| 端口 | 状态 | 用途 | 安全等级 |
|------|------|------|---------|
| 22 | 开放 | SSH | ✅ 必要 |

**安全等级：** ✅ 优秀

## 第七部分：速率限制

### 7.1 什么是速率限制？

**速率限制 (Rate Limiting)** 限制每个 IP 的请求频率，防止：
- DDoS 攻击
- 暴力破解
- API 滥用

---

### 7.2 我们的速率限制配置

**Nginx 配置（建议添加）：**
```nginx
# 在 http 块中定义限流区域
http {
    # 生产环境：100 请求/分钟
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=100r/m;
    
    # 测试环境：500 请求/分钟
    limit_req_zone $binary_remote_addr zone=test_limit:10m rate=500r/m;
}

# 在 location 块中应用
location /api/ {
    limit_req zone=api_limit burst=20 nodelay;
    proxy_pass http://127.0.0.1:8001;
}
```

**参数说明：**
- `rate=100r/m` - 每分钟 100 个请求
- `burst=20` - 允许突发 20 个请求
- `nodelay` - 立即处理突发请求

---

### 7.3 FastAPI 内置限流（可选）

使用 `slowapi` 库：

```python
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

@app.get("/api/v1/some-endpoint")
@limiter.limit("100/minute")
async def some_endpoint(request: Request):
    return {"message": "OK"}
```

## 第八部分：认证与授权

### 8.1 JWT 认证

**JWT (JSON Web Token)** 用于用户认证。

**工作流程：**
```
1. 用户登录 → 发送用户名密码
2. 服务器验证 → 返回 JWT
3. 客户端存储 JWT
4. 后续请求携带 JWT
5. 服务器验证 JWT → 允许访问
```

**JWT 结构：**
```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.    # Header (算法)
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.  # Payload (数据)
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  # Signature (签名)
```

---

### 8.2 我们的 JWT 配置

**Secrets 文件 (/etc/ai-study/prod.secrets)：**
```bash
# JWT 密钥（至少 32 字符）
JWT_SECRET=your-random-jwt-secret-key-at-least-32-chars

# JWT 过期时间（秒）
JWT_EXPIRATION=86400  # 24 小时
```

**安全建议：**
1. ✅ 使用强随机密钥（至少 32 字节）
2. ✅ 设置合理的过期时间
3. ✅ 定期轮换密钥
4. ❌ 不要硬编码在代码中
5. ❌ 不要提交到 Git
